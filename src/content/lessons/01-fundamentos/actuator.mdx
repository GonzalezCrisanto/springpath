Tu app arranca, los tests pasan, el deploy salió bien. Y entonces — inevitablemente — algo falla en producción. La pregunta es: ¿tenés forma de entender qué pasó?

¿Está respondiendo la app? ¿Conecta a la base de datos? ¿Cuánta memoria JVM está usando? ¿El pool de conexiones se está llenando? Sin respuestas a esas preguntas, operar una aplicación es volar a ciegas — y cuando algo falla, lo único que podés hacer es adivinar.

**Spring Boot Actuator** resuelve eso de raíz. Con agregar una dependencia, tu app expone un conjunto de endpoints HTTP que muestran su estado interno: salud, métricas, configuración activa, loggers. No instrumentás nada vos mismo — está todo ahí.

## Por qué esto importa más allá de la curiosidad

El problema no es solo que vos quieras saber qué pasa. Los sistemas que corren tu app en producción también necesitan saberlo:

- **Kubernetes** hace chequeos periódicos a un endpoint de salud para decidir si reinicia un pod o lo saca de rotación. Si tu app no responde nada útil, el orquestador reinicia el pod — aunque la app esté técnicamente viva.
- **Prometheus** necesita un endpoint para scrapear métricas cada N segundos y alimentar los dashboards de Grafana.
- **Tu equipo** necesita saber si la DB conecta, si el disco se está llenando, o si el pool de conexiones HikariCP está saturado — antes de que lo reporten los usuarios.

Sin Actuator, tu app es una caja negra para todos ellos.

## Habilitarlo

Agregás el starter:

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

Por defecto, Actuator expone solo `/actuator/health` e `/actuator/info` por HTTP. El resto existe internamente pero no es accesible desde la web — ya vas a ver por qué eso es intencional.

Para habilitar más:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,env,loggers
  endpoint:
    health:
      show-details: always  # muestra el estado de cada componente, no solo UP/DOWN global
```

## Los endpoints que vas a usar siempre

### `/actuator/health`

El más importante. Devuelve el estado de la app y de cada componente integrado. Con `show-details: always`:

```json
{
  "status": "UP",
  "components": {
    "db": {
      "status": "UP",
      "details": {
        "database": "PostgreSQL",
        "validationQuery": "isValid()"
      }
    },
    "diskSpace": {
      "status": "UP",
      "details": {
        "total": 250685575168,
        "free": 89432014848,
        "threshold": 10485760
      }
    }
  }
}
```

El `status` en la raíz agrega todos los componentes: si cualquiera devuelve `DOWN`, el estado global también es `DOWN`. Kubernetes basa sus decisiones de reinicio en ese valor.

### `/actuator/metrics`

<Term definition="Valores numéricos que describen el comportamiento de la app en el tiempo: requests por segundo, latencia, uso de memoria JVM, conexiones activas al pool de DB, etc.">Métricas</Term> numéricas recolectadas sin configuración extra. Primero pedís el listado:

```
GET /actuator/metrics
```

Y después el detalle de cada una por nombre:

```
GET /actuator/metrics/http.server.requests
GET /actuator/metrics/jvm.memory.used
GET /actuator/metrics/hikaricp.connections.active
```

Tenés métricas de JVM, Tomcat, HikariCP y de tus endpoints HTTP — todo gratis por el simple hecho de tener el starter.

### `/actuator/info` y `/actuator/env`

`/info` muestra metadata estática que configurás vos en `application.yml`:

```yaml
info:
  app:
    name: Springpath
    version: 1.0.0
```

`/env` muestra todas las propiedades activas — útil para diagnosticar por qué una configuración no tiene el valor que esperás. Fijate que también muestra variables de entorno y propiedades del sistema.

## Agregar tu propio indicador de salud

¿Tu app depende de un servicio externo — una API de pagos, un sistema legado? Podés agregar ese chequeo implementando <Term definition="Interfaz de Spring que permite registrar un componente de salud personalizado. Spring lo detecta automáticamente y lo incluye en /actuator/health.">HealthIndicator</Term>:

```java
@Component
public class ExternalApiHealthIndicator implements HealthIndicator {

    private final PaymentGatewayClient client;

    public ExternalApiHealthIndicator(PaymentGatewayClient client) {
        this.client = client;
    }

    @Override
    public Health health() {
        try {
            client.ping();
            return Health.up()
                .withDetail("gateway", "Stripe")
                .withDetail("status", "reachable")
                .build();
        } catch (Exception e) {
            return Health.down()
                .withDetail("gateway", "Stripe")
                .withDetail("error", e.getMessage())
                .build();
        }
    }
}
```

Spring detecta automáticamente cualquier bean que implemente `HealthIndicator` y lo suma a la respuesta de `/actuator/health`. No registrás nada en ningún lado — es el mismo mecanismo de autodescubrimiento de siempre.

## Por qué no exponés todo en producción

Acá viene algo importante que mucha gente ignora: algunos endpoints de Actuator son peligrosos si quedan expuestos sin protección.

`/actuator/env` muestra todas tus propiedades — incluidas contraseñas y claves de API si no están enmascaradas. `/actuator/shutdown` puede apagar la app remotamente. `/actuator/loggers` permite cambiar el nivel de log en runtime.

La práctica estándar tiene dos partes:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics  # solo lo que el sistema de monitoreo necesita
  server:
    port: 8081  # Actuator en un puerto separado, inaccesible desde internet
```

Con el puerto separado, configurás el firewall para que solo los sistemas internos de monitoreo accedan al `8081`. Los usuarios de la app solo ven el `8080`. Un endpoint de diagnóstico nunca debería estar en la misma superficie que la API pública.

## El siguiente paso: Prometheus y Grafana

Actuator es el primer eslabón. El stack completo es:

**Actuator** → recolecta métricas internamente  
**Micrometer** → las normaliza en un formato común  
**Prometheus** → las scrapeá y las almacena  
**Grafana** → las muestra en dashboards

Con solo agregar `micrometer-registry-prometheus`, Actuator empieza a exportar todo en el formato que Prometheus espera en `/actuator/prometheus`. Ese flujo completo lo vas a ver en el Track 6.
