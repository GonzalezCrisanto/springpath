Antes de Spring MVC, construir una API en Java significaba extender `HttpServlet` y gestionar el ruteo a mano. Algo así:

```java
@Override
protected void service(HttpServletRequest req, HttpServletResponse res) {
    String method = req.getMethod();
    String path   = req.getPathInfo();

    if ("GET".equals(method) && "/products".equals(path)) {
        // listar productos
    } else if ("GET".equals(method) && path.matches("/products/\\d+")) {
        // obtener uno
    } else if ("POST".equals(method) && "/products".equals(path)) {
        // crear
    } // ... y así para cada endpoint
}
```

¿Lo ves? Un `if` por cada combinación de método + path. Con tres endpoints ya es incómodo. Con cincuenta, es un quilombo. Y el ruteo es solo el primer problema — todavía tenés que parsear el body, convertir parámetros, serializar la respuesta…

Spring MVC resuelve todo eso con un patrón que se llama <Term definition="Patrón de diseño donde un único punto de entrada (el DispatcherServlet) recibe todos los requests y los delega al handler correcto. Elimina el ruteo manual y centraliza preocupaciones transversales como autenticación y manejo de errores.">Front Controller</Term>: un único servlet — el `DispatcherServlet` — que recibe todos los requests y los despacha al método correcto según las anotaciones. Sin un solo `if`.

## `@RestController` — declarar que esta clase maneja requests

Un controller es una clase anotada con `@RestController`. Eso combina dos cosas:

```java
@RestController
public class ProductController {

    @GetMapping("/products")
    public List<Product> listAll() {
        return productService.findAll(); // Spring serializa esto a JSON automáticamente
    }
}
```

- `@Controller` registra la clase en el contenedor como handler de requests HTTP.
- `@ResponseBody` le dice a Spring que el valor de retorno se escribe directamente en el cuerpo de la respuesta — no es el nombre de una vista a renderizar.

`@RestController` es la combinación de las dos. La usás siempre que estés construyendo una API REST. `@Controller` solo lo vas a ver cuando renderizás templates del lado del servidor (Thymeleaf, por ejemplo).

## `@RequestMapping` — conectar un método con un path y un verbo

La anotación `@RequestMapping` es la que hace el trabajo. Puede ir a nivel de clase para un prefijo compartido, y a nivel de método para el sub-path y el verbo:

```java
@RestController
@RequestMapping("/api/products")     // prefijo compartido por todos los métodos
public class ProductController {

    @GetMapping                      // GET  /api/products
    public List<Product> listAll() { … }

    @GetMapping("/{id}")             // GET  /api/products/42
    public Product getById(@PathVariable Long id) { … }

    @PostMapping                     // POST /api/products
    @ResponseStatus(HttpStatus.CREATED)
    public Product create(@RequestBody Product product) { … }

    @PutMapping("/{id}")             // PUT  /api/products/42
    public Product update(@PathVariable Long id, @RequestBody Product product) { … }

    @DeleteMapping("/{id}")          // DELETE /api/products/42
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void delete(@PathVariable Long id) { … }
}
```

Fijate en los shortcuts: `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping`. Son equivalentes a `@RequestMapping(method = RequestMethod.GET/POST/…)` pero mucho más legibles. Son la forma recomendada en la doc oficial — usá siempre los shortcuts.

## La semántica de los verbos HTTP

El protocolo define verbos con significado preciso. En una API bien diseñada esa semántica no es opcional — es parte del contrato con los clientes:

| Verbo | Uso | ¿Idempotente? | ¿Lleva body? |
|-------|-----|:-:|:-:|
| `GET` | Leer un recurso | ✅ | No |
| `POST` | Crear un recurso | ❌ | ✅ |
| `PUT` | Reemplazar un recurso completo | ✅ | ✅ |
| `PATCH` | Actualizar parcialmente | ❌ | ✅ |
| `DELETE` | Eliminar un recurso | ✅ | No |

¿Qué significa <Term definition="Una operación es idempotente si ejecutarla N veces tiene el mismo efecto que ejecutarla una sola vez. GET, PUT y DELETE son idempotentes. POST no lo es: dos POSTs iguales pueden crear dos recursos distintos.">idempotente</Term>? Que repetir la operación no cambia el resultado. `DELETE /products/42` dos veces deja el recurso igual de eliminado. `POST /products` dos veces crea dos productos distintos.

Eso importa para los clientes: si un POST falla y no saben si llegó al servidor, ¿lo reintentán? Con un PUT, sí pueden — con un POST, tienen que chequear primero.

## Extraer datos del request

### `@PathVariable` — variables en el path

```java
// GET /products/42  →  id = 42
@GetMapping("/{id}")
public Product getById(@PathVariable Long id) {
    return productService.findById(id);
}
```

Spring extrae el segmento `42` de la URL y lo convierte al tipo `Long` declarado en el parámetro. Si no puede convertir — por ejemplo, `GET /products/abc` — lanza un `400 Bad Request` automáticamente. No tenés que validar eso vos.

### `@RequestParam` — query parameters

```java
// GET /products?category=electronics&page=0&size=20
@GetMapping
public Page<Product> list(
    @RequestParam String category,
    @RequestParam(defaultValue = "0") int page,
    @RequestParam(defaultValue = "20") int size
) { … }
```

`defaultValue` hace el parámetro opcional — si no viene en el request, usa ese valor. Sin `defaultValue`, el parámetro es requerido y Spring devuelve `400` si falta.

### `@RequestBody` — el cuerpo del request

```java
@PostMapping
@ResponseStatus(HttpStatus.CREATED)
public Product create(@RequestBody Product product) {
    return productService.save(product);
}
```

Spring usa <Term definition="Librería de serialización JSON incluida por defecto en spring-boot-starter-web. Convierte automáticamente JSON a objetos Java en el request, y objetos Java a JSON en la respuesta.">Jackson</Term> para deserializar el JSON del body al tipo Java que declaraste. No escribís un solo `objectMapper.readValue(…)` — Spring lo hace por vos.

## Control total con `ResponseEntity`

Cuando necesitás controlar el status HTTP o los headers, usás `ResponseEntity`:

```java
@GetMapping("/{id}")
public ResponseEntity<Product> getById(@PathVariable Long id) {
    return productService.findById(id)
        .map(ResponseEntity::ok)                 // 200 con el body
        .orElse(ResponseEntity.notFound().build()); // 404 sin body
}

@PostMapping
public ResponseEntity<Product> create(@RequestBody Product product) {
    Product saved = productService.save(product);
    URI location = URI.create("/api/products/" + saved.getId());
    return ResponseEntity
        .created(location)   // 201 Created con el header Location apuntando al nuevo recurso
        .body(saved);
}
```

Cuando retornás el objeto directamente, Spring asume `200 OK`. `ResponseEntity` lo usás cuando ese default no alcanza: un `POST` que debe retornar `201 Created`, un `GET` que puede retornar `404`, un endpoint que necesita agregar headers de caché.
